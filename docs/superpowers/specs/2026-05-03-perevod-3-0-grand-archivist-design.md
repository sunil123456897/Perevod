# Спецификация дизайна: Perevod 3.0 "Пайплайн Великого Архивариуса"

**Дата:** 2026-05-03  
**Версия:** 1.0  
**Статус:** Draft  
**Цель:** Описать архитектуру многоагентной системы перевода с глубокой интеграцией RAG и самокоррекцией.

---

## 1. Обзор (Overview)
Perevod 3.0 переходит от линейного перевода к циклическому агентному пайплайну. Система имитирует работу профессионального издательства: сбор контекста -> анализ -> перевод -> критика -> редактура -> архивация знаний.

### Основные агенты:
1.  **Chronicle Agent (Летописец):** Управляет Библией Вселенной (World Bible).
2.  **Translator Agent (Переводчик):** Создает черновой литературный перевод.
3.  **Judge Agent (Судья):** Критикует перевод на основе словаря, стиля и логики.
4.  **Editor Agent (Редактор):** Исправляет текст на основе замечаний Судьи.

---

## 2. Адаптивная архитектура (LangGraph)

### Узлы (Nodes):
1.  **`context_retrieval`**: (Cache Key: `project_name + last_3_chapters_hash`) Загрузка Библии и Rolling Memory.
2.  **`pre_analysis`**: (Условно/Кэшируемо) Летописец ищет новые сущности. Пропускается, если есть в кэше.
3.  **`translate`**: (Cache Key: `chapter_text + context_hash`) Основной переводчик.
4.  **`judge`**: Оценка по критериям Blocking Issues.
5.  **`refine`**: (Условно) Вызывается только при `blocking_issues = true`.
6.  **`post_analysis`**: Обновление Библии (статус `candidate`).

### Определения Blocking Issues:
- Пропуск абзацев или предложений.
- Использование неверного перевода для терминов из словаря.
- Ошибка в поле/имени ключевого персонажа (из Библии).
- Сломанная структура JSON/Markdown.
- Незавершенные фразы в конце главы.

## 4. Структура данных (State)

Обновить `AgentState` в `src/Perevod/agents/state.py`:
```python
class AgentState(TypedDict):
    # ... existing ...
    rag_context: List[str]          # Данные из Библии и памяти
    chapter_memory: List[str]       # Краткое содержание последних 3 глав
    current_draft: str              # Текущий черновик главы
    judge_result: Dict[str, Any]    # Полный ответ Судьи
    blocking_issues: List[str]      # Список критических ошибок для Редактора
    refinement_count: int           # Счетчик итераций (max 2)
    final_translation: str          # Финальный текст
    cache_keys: Dict[str, str]      # Хеши для каждого этапа
```

---

## 5. Стратегия внедрения (MVP)

1.  **Этап 1 (MVP):** Реализация цикла `translate -> judge -> (if issues) refine -> save`. Без активного изменения Библии.
2.  **Этап 2:** Интеграция RAG (Rolling Memory) и кэширования этапов.
3.  **Этап 3:** `pre_analysis` и `post_analysis` (система кандидатов).
