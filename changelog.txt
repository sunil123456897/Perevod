# Changelog

## 2026-05-24 - Workflow stage correctness

- Context retrieval now builds chapter-specific RAG context for every chapter in a batch instead of reusing context from the first chapter.
- Translation now uses the chapter-specific context when building prompts and cache keys, while keeping the existing global context as a fallback.
- Analysis results now carry `source_chapter` metadata so reports can identify which chapter reached the analysis stage.
- Refine now marks only edited chapters as `refined` and returns updated `processed_chapters` for downstream reporting.
- `translation_report.json` now includes per-chapter `context`, `analysis`, and `refine` stage statuses in addition to translation, judge, and summary.
- Verification: `python -m ruff check src scripts tests` and `python scripts\run_safe_tests.py` both pass.

## 2026-05-24 - SQLite chapter checkpoints

- Added a project-scoped `chapter_runs` SQLite table for structured chapter checkpoint state.
- Added `DatabaseManager` APIs for discovering chapters, marking individual stages, recording errors, and reading chapter runs.
- Workflow startup now records each processable chapter as `discovered`.
- Context retrieval, analysis, glossary update, translation, output write, judge, refine, summary, and memory update stages now write best-effort per-chapter checkpoints.
- Checkpoint write failures are warning-only so a transient SQLite problem does not falsely invalidate an already written translation.
- Verification: `python -m ruff check src scripts tests`, `python scripts\run_safe_tests.py`, and `python -m compileall -q src scripts` all pass.

## 2026-05-24 - Retry planning from SQLite checkpoints

- `--retry-failed` and `--retry-incomplete` can now recover retry candidates from SQLite `chapter_runs` when `translation_report.json` is missing.
- Downstream failures after `output_written` now mark the chapter for retry while setting `reuse_existing_translation`, so the workflow can continue judge/refine/summary without spending translation API again.
- Existing strict report validation is preserved: malformed reports still fail instead of being silently ignored.
- Verification: targeted graph/checkpoint tests and `python -m ruff check src scripts tests` pass.

## 2026-05-24 - Stage gates from SQLite checkpoints

- Graph state now carries SQLite `chapter_runs`, allowing nodes to inspect completed stages during retry/resume.
- Analysis skips chapters with `analysis_done=done`, avoiding unnecessary analysis model calls.
- Judge skips chapters with `judge_done=done` and emits a checkpoint-backed passing result for downstream reporting.
- Summarization skips chapters with both `summary_done=done` and `memory_updated=done`, avoiding duplicate summary model calls and memory writes.
- Verification: targeted node/graph checkpoint tests and ruff pass.

## 2026-05-24 - Additional safe stage gates

- Context retrieval can reuse an already available chapter context when `context_retrieved=done`, avoiding duplicate retrieval without dropping context.
- Glossary curation skips when every chapter in the current batch has `glossary_updated=done`, relying on the SQLite-backed project dictionary as the source of truth.
- Refine skips chapters with `refine_done=done`, marks the processed chapter as checkpoint-reused, and avoids duplicate editor model calls.
- Verification: targeted node/graph checkpoint tests and ruff pass.

## 2026-05-24 - Persisted chapter context checkpoints

- `chapter_runs` now stores the chapter-specific retrieved context, allowing context resume across separate process runs.
- `DatabaseManager.update_chapter_context()` persists context after successful retrieval and `get_chapter_runs()` returns it with checkpoint state.
- Context retrieval now restores persisted context when `context_retrieved=done`, avoiding duplicate retrieval without losing the context needed by translation.
- Verification: targeted retrieval/database/graph tests and ruff pass.

## 2026-05-24 - Legacy SQLite checkpoint repair

- Database startup now repairs legacy `chapter_runs` tables that predate the persisted `context` column.
- The repair is intentionally narrow: it only adds `chapter_runs.context TEXT` when the table exists and the column is missing.
- Verification: targeted legacy-schema/database tests and ruff pass.

## 2026-05-24 - Report stages from SQLite checkpoints

- `translation_report.json` now merges per-chapter SQLite checkpoint evidence into stage statuses.
- Checkpoint-backed reports can show `glossary`, `output`, and `memory` stages in addition to context, analysis, translation, judge, refine, and summary.
- Workflow report writing now refreshes `chapter_runs` from SQLite after graph execution, so node-written checkpoints are reflected in the final report instead of the startup snapshot.
- Refine/editor failures now mark only the `refine` stage as failed in `translation_report.json` instead of also downgrading an already completed `judge` stage.
- Summary failures now fail the workflow and mark the affected chapter as failed in `translation_report.json`, while preserving the already completed translation/judge stages for safe retry.
- Report status resolution is now stage-aware for all chapters: any chapter with a `failed` stage is marked `failed`, not only the first chapter parsed from the top-level workflow error text.
- Failed chapters now surface chapter-specific checkpoint errors in `translation_report.json` (`chapter.error`) when the failure comes from stage checkpoints rather than the single top-level workflow error text.
- Chapter-level analysis/context retrieval failures are now treated as downstream workflow failures (`RuntimeError`) and reported with `analysis/context = failed` plus `status = failed` for the affected chapters.
- Global context retrieval failures (`context_errors` with `title="*"`) are now treated as downstream workflow failures as well, with `context = failed` and `status = failed` for affected report chapters instead of warning-only success.
- `retry_failed` now reuses existing translated output for failed chapters caused by downstream analysis/context/retrieval stages, avoiding unnecessary translation API calls after non-translation failures.
- Stage merging now preserves current-run failures over stale checkpoint `done`: runtime `context/analysis/summary` errors re-apply `failed` after checkpoint stage merge so report stages cannot be silently downgraded to `done`.
- Stage inference now has an error-text fallback for downstream failures (`Analysis failed...`, `Context retrieval failed...`, `Summary failed...`) so the corresponding chapter stage remains `failed` even if explicit per-stage error arrays are missing.
- Report `stages` schema is now consistent for non-skipped chapters with explicit `glossary`, `output`, and `memory` keys alongside `context/analysis/translation/judge/refine/summary`, improving deterministic parsing and diagnostics.
- Failed chapters without a detailed error payload now receive a deterministic fallback `chapter.error` derived from the first failed stage (for example, `Failed stage: judge`) instead of `null`.
- `retry_incomplete` now treats stage `not_started`/non-`done` states as incomplete even without warnings, with backward-compatible handling for legacy reports that only have `translation/judge/summary` stage keys.
- `retry_incomplete` now explicitly ignores report chapters with `status` starting with `skipped`, preventing false-positive retries of previously skipped chapters whose stages are `skipped`.
- `retry_incomplete` stage checks are now adaptive to hybrid report schemas: only stage keys that are actually present are required, so missing newer keys (`glossary/output/memory`) in older reports no longer trigger false-positive retries.
- SQLite-checkpoint retry selection now mirrors report logic for `skipped*` statuses: `_should_retry_checkpoint_run` ignores skipped chapter runs, preventing false-positive `retry_incomplete` processing when report file is missing.
- SQLite-checkpoint incomplete checks are now adaptive to hybrid checkpoint schemas: missing newer checkpoint keys no longer trigger false-positive retries; only present stage keys are required, with legacy fallback to `translation/judge/summary`.
- SQLite-checkpoint retry adapters now preserve `glossary` and `output` stage evidence, so `output_written=failed` cannot be lost when retrying without `translation_report.json`.
- Report chapter status now resolves to `translated` when checkpoint stages confirm `translation=done` even if `processed_chapters` is empty in final state, preventing false `pending` statuses after checkpoint-driven resumes.
- Retry planning no longer reuses an existing output file when report/checkpoint stage evidence says `output=failed`, preventing a stale or partial file from being treated as safe translated output.
- Verification: graph runner/database/retrieval targeted tests, `python -m ruff check src scripts tests`, `python -m compileall -q src scripts`, and `python scripts\run_safe_tests.py` pass.

## 2026-05-24 - Resume-safe judge/refine continuity

- Fixed report counter correctness for checkpoint-driven resumes: when runtime `processed_chapters` is empty, `processed_count/processed_chapters` now fall back to chapters with `translation=done` in stage evidence.
- Added checkpoint retry safety for judge/refine continuity: retry chapters loaded from SQLite checkpoints now set `force_rejudge` when `judge=done` but judge blocking details are unavailable, preventing silent skip of required refine.
- `judge_node` now honors `force_rejudge`: it re-runs judge instead of blindly skipping on `judge_done`, preserving downstream refine correctness without redoing translation.
- Retry planning from `translation_report.json` now preserves known chapter `blocking_issues` when reusing existing translation output, so downstream refine requirements are not silently dropped.
- `judge_node` checkpoint-skip path no longer clears known chapter blockers: when `blocking_issues` are already present in chapter state, they are kept and propagated to `judge_results`/global `blocking_issues`.
- Report summary-stage inference now ignores retrieval-memory entries in `chapter_summaries` (e.g. `{"content", "chapter_index", "title"}`) and only treats real summarization artifacts (`summary/key_events/active_characters`) or checkpoint reuse as `summary=done`.
- Translation context fallback is now chapter-safe: in multi-chapter runs, a chapter without its own `chapter_contexts[title]` no longer receives global `rag_context` (first-chapter context), preventing cross-chapter context leakage.
- SQLite `chapter_runs` now stores structured judge artifacts in `judge_result_json` (with legacy schema auto-repair), and `DatabaseManager.get_chapter_runs()` exposes them as `judge_result`.
- `judge_node` now persists per-chapter judge payloads to SQLite after successful evaluation and can reuse checkpointed `blocking_issues/pass_check/suggestions/severity/score` when `judge_done=done`.
- Retry planning from SQLite checkpoints now carries checkpointed `blocking_issues` through `_checkpoint_run_to_report_chapter`, preventing forced rejudge when judge evidence is already available and preserving chapter-level refine requirements.
- Report status/QA fields now consume checkpointed `judge_result` as fallback when runtime `judge_results` are missing: checkpoint-only chapters with `translation=done` + blocking issues are reported as `qa_failed` (not false `translated`) with `judge_pass_check/judge_score/judge_severity/quality_suggestions`.
- SQLite `chapter_runs` now stores structured refine artifacts in `refine_result_json` (with legacy schema auto-repair), and `DatabaseManager.get_chapter_runs()` exposes them as `refine_result`.
- `refine_node` now persists per-chapter refine payloads after successful correction and restores checkpointed refine metadata when `refine_done=done`, without re-calling the editor model.
- `translation_report.json` now exposes refine metadata (`refined`, `refine_checkpoint_reused`, `refinement_count`, `refine_issues_fixed`) using runtime or checkpoint evidence, improving post-failure diagnostics and resume visibility.
- SQLite `chapter_runs` now stores structured summary artifacts in `summary_result_json` (with legacy schema auto-repair), and `DatabaseManager.get_chapter_runs()` exposes them as `summary_result`.
- `summarization_node` now persists per-chapter summary payloads including `chapter_index` and restores checkpointed summary metadata when `summary_done/memory_updated=done`, without re-calling the summarization model.
- `translation_report.json` now exposes summary metadata (`summary_checkpoint_reused`, `summary`, `summary_key_events`, `summary_active_characters`, `summary_chapter_index`) using runtime or checkpoint evidence.
- SQLite-only retry selection now treats checkpointed `judge_result.blocking_issues` as retry-worthy QA blockers even when `chapter_runs.status` is still a stage name such as `judge_done`; the checkpoint-to-report adapter also normalizes those chapters to `qa_failed`.
- Retry planning now forces re-judge when a checkpoint has stale judge blockers and `refine=done`: after editor correction, the old judge payload is no longer treated as final evidence, preventing retry loops or false QA failure after a crash before re-judge.
- Report-based `--retry-failed` now treats non-empty chapter `blocking_issues` as retry-worthy even if a legacy or partially inconsistent report still says `status=translated`, preventing QA-blocked chapters from being silently skipped.
- `refine_node` no longer writes editor-corrected text to the translation cache before the follow-up judge pass approves it; QA-approved caching remains centralized in `judge_node`.
- Translation cache keys now include an explicit translation prompt version and normalized generation settings (`temperature`, `top_p`) in addition to model/input/dictionary/context/style data, so prompt or sampling changes invalidate stale cache entries.

### Архитектура и Целостность Данных
- **Упрощена архитектура перевода:** Основной граф теперь использует более простую и надежную стратегию перевода **целыми главами**. Это устраняет сложность, связанную с семантическим чанкингом.
- **Повышена отказоустойчивость удаления:** Логика удаления записей в GUI (Словарь, Библия) была переработана. Теперь операции с векторной (ChromaDB) и реляционной (SQLite) базами данных обернуты в отдельные блоки try-except, чтобы предотвратить рассинхронизацию данных при сбоях.

### Качество Кода и Зависимости
- **Удален неиспользуемый код:** Компонент `SemanticChunker` и его зависимость `nltk` были полностью удалены из проекта, так как они не используются в текущей архитектуре.
- **Устранен анти-паттерн `sys.path.append`:** Все скрипты в папке `scripts` были исправлены для корректной работы в качестве модулей Python, что соответствует лучшим практикам и повышает надежность их запуска.
- **Исправлены комментарии и документация:** Комментарии в коде и данный changelog были обновлены, чтобы точно отражать текущую функциональность, в частности, отказ от перевода по чанкам в пользу перевода целыми главами.
